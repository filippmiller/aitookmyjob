# Платформа для людей, потерявших работу из-за ИИ
## Результаты исследования рынка + концепция сайта + 30 сценариев пользователей

---

## ЧАСТЬ 1: РЕЗУЛЬТАТЫ ИССЛЕДОВАНИЯ

### Что уже существует на рынке

На сегодня (февраль 2026) существует несколько ресурсов, связанных с темой потери работы из-за ИИ, но **ни один из них не является полноценной community-платформой** с интерактивом, тегами по компаниям, статистикой и группами поддержки. Вот ключевые игроки:

#### 1. aitookmyjob.io — ближайший аналог
- **Формат:** Простой сайт-каталог пользовательских историй, разбитых по профессиям (копирайтер, дизайнер, разработчик и т.д.)
- **Контент:** Всего около 9 категорий и менее 15 историй
- **Слабости:** Нет сообщества, нет комментариев, нет статистики по компаниям, нет тегов, нет форума, нет группы поддержки. По сути, это лендинг с формой и карточками историй
- **Вывод:** Проект существует, но крайне сырой и неразвитый. Это ваш прямой конкурент, но его легко превзойти

#### 2. Blood in the Machine — серия «AI Killed My Job» (Брайан Мерчант)
- **Формат:** Substack-рассылка журналиста Брайана Мерчанта, десятки тысяч подписчиков
- **Контент:** Серия глубоких статей с реальными историями работников. Выпуски по индустриям: tech workers, translators, copywriters, artists
- **Слабости:** Это журналистский проект одного автора, а не платформа. Нет интерактива, нет базы данных. Истории отбирает и редактирует журналист, а не публикует сообщество
- **Вывод:** Доказывает огромный спрос на тему. Мерчант получил «сотни» откликов после первого поста

#### 3. Трекеры увольнений (без фокуса на ИИ-историях)
- **ailayofftracker.com** — дашборд по AI-увольнениям, но без пользовательских историй
- **ailayoffs.com** — трекер, но с заглушками данных (mock values)
- **layoffs.fyi / trueup.io** — крупные трекеры tech-увольнений (783 events в 2025), но без фильтра «из-за ИИ» и без историй
- **warntracker.com** — трекер по WARN Act, формальные данные
- **skillsyncer.com** — общий трекер увольнений с аналитикой
- **jobripper.ai** — AI-трекер увольнений по секторам

#### 4. Reddit и медиа-агрегаторы
- Вирусные треды на r/AskReddit собирают тысячи откликов
- BuzzFeed, Bored Panda, Yahoo публикуют подборки историй из Reddit
- Но всё это разрозненно — нет единого хранилища, нет тегов, нет статистики

### Вывод: рыночная ниша открыта

**Не существует ни одной платформы, которая бы объединяла:**
1. Пользовательские истории с тегами по компаниям, профессиям, датам
2. Интерактивное сообщество (комментарии, группы, поддержка)
3. Живую статистику и аналитику по AI-увольнениям
4. Раздел «что делать дальше» (переквалификация, ресурсы)
5. Возможность отслеживать компании-«рецидивистов»

Ваша идея — это незанятая ниша между журналистикой (Blood in the Machine), трекерами (layoffs.fyi) и Reddit.

## ЧАСТЬ 2: КОНЦЕПЦИЯ ПЛАТФОРМЫ

### Рабочее название
**«AI Took My Career»** / **«Уволены ИИ»** / **«The Human Cost»**

### Структура сайта

#### Главные разделы:
1. **Истории** — ядро платформы, пользовательские истории с фильтрами
2. **Компании** — профили компаний с рейтингами и статистикой увольнений
3. **Статистика** — живые дашборды и аналитика
4. **Сообщество** — группы по профессиям, форумы, взаимопомощь
5. **Ресурсы** — переквалификация, юридическая помощь, вакансии
6. **Новости** — агрегатор новостей про AI и рынок труда

### Форма для подачи истории (ключевые поля):

| Поле | Тип | Обязательное |
|------|-----|-------------|
| Имя / псевдоним | Текст | Да |
| Профессия | Выпадающий список + своё | Да |
| Компания | Текст с автоподсказкой | Нет (можно анонимно) |
| Страна / город | Геолокация | Да |
| Дата увольнения | Дата | Да |
| Стаж в компании | Число лет | Да |
| Зарплата до увольнения | Диапазон | Нет |
| Как уволили | Выбор: массовое/индивидуальное/сокращение/контракт не продлили | Да |
| Причина (как объяснила компания) | Текст | Да |
| Какой ИИ-инструмент заменил | Текст | Нет |
| Предупредили ли заранее | Да/Нет/Частично | Да |
| Получили ли компенсацию | Да/Нет + детали | Нет |
| Нашли ли новую работу | Да/Нет/Ищу | Да |
| Если да — в какой сфере | Текст | Нет |
| Время поиска новой работы | Число месяцев | Нет |
| Ваша история | Развёрнутый текст | Да |
| Совет другим | Текст | Нет |
| Настроение | Шкала 1-10 | Нет |

### Профиль компании (автоматический):
- Название, логотип, отрасль
- Количество историй с упоминанием
- Средний стаж уволенных
- Какие профессии затронуты
- Временная шкала увольнений
- «Рейтинг гуманности» (на основе компенсаций, предупреждений и т.д.)
- Вернулась ли компания к найму людей

### Группы и подгруппы сообщества:

**По профессиям:**
- Копирайтеры и контент-менеджеры
- Графические дизайнеры и иллюстраторы
- Разработчики и QA-инженеры
- Переводчики
- Голосовые актёры и музыканты
- Маркетологи
- HR-специалисты
- Журналисты и редакторы
- Юристы и парала-галы
- Бухгалтеры и финансисты
- Учителя и тьюторы
- Операторы call-центров
- Фотографы и видеографы

**По ситуации:**
- «Только что уволили» — группа поддержки
- «Ищу работу» — обмен вакансиями
- «Переквалифицировался» — истории успеха
- «Фрилансер без клиентов» — для тех, чей рынок обвалился
- «AI вернули людей назад» — когда компания наняла обратно

**По географии:**
- США, Европа, СНГ, Азия и т.д.

## ЧАСТЬ 3: СТАТИСТИКА И АНАЛИТИКА

### Автоматические дашборды:

1. **Топ компаний по количеству AI-увольнений** — ранжированный список
2. **Динамика увольнений по месяцам** — график с трендом
3. **Карта мира** — геолокация увольнений
4. **Наиболее затронутые профессии** — pie chart
5. **Среднее время поиска новой работы** — по профессиям
6. **Процент нашедших работу vs. не нашедших** — по профессиям
7. **Какие ИИ-инструменты чаще всего заменяют людей** — рейтинг (ChatGPT, Midjourney, Copilot и т.д.)
8. **Уровень компенсации** — средний размер выходного пособия
9. **Сколько компаний наняли людей обратно** — показатель «отката»
10. **Тренд зарплат** — до и после переквалификации

### Метрики для каждой истории:
- Количество просмотров
- Количество «мне тоже так» (аналог лайка)
- Количество комментариев
- Отметка «обновлено» (человек нашёл работу)

## ЧАСТЬ 4: 30 СЦЕНАРИЕВ ПОЛЬЗОВАТЕЛЕЙ

Каждый сценарий описывает: кто этот человек, какую работу потерял, почему, что чувствует, чем хотел бы поделиться на платформе и какой интерактив ему нужен.

---

### Сценарий 1: Маша, копирайтер, 28 лет, Москва
**Компания:** Маркетинговое агентство (15 человек)
**Что случилось:** Клиенты начали присылать тексты, сгенерированные ChatGPT, с просьбой «просто поправить». Ставки упали в 5 раз. Агентство закрылось.
**Хочет поделиться:** Как рынок копирайтинга обвалился за 18 месяцев. Хочет найти тех, кто прошёл через то же самое.
**Нужный интерактив:** Группа «Копирайтеры», форум «Куда уходить из текстов», раздел «Кто переучился».
**Нашла ли работу:** Учится на UX-райтера, пока подрабатывает барменом.

---

### Сценарий 2: Джейсон, графический дизайнер, 34 года, Чикаго
**Компания:** Средний e-commerce бренд
**Что случилось:** Босс показал Midjourney-макет клиенту раньше, чем работу дизайнерской команды. Клиент выбрал AI. Команду из 4 человек сократили до 1.
**Хочет поделиться:** Как обидно, когда твоя работа обесценивается AI-слопом. Хочет предупредить коллег.
**Нужный интерактив:** Возможность оценить профиль компании, поставить «красный флаг».
**Нашёл ли работу:** Перешёл в 3D-моделирование для игр, нашёл работу через 7 месяцев.

---

### Сценарий 3: Елена, переводчик (EN-RU), 42 года, Санкт-Петербург
**Компания:** Крупное бюро переводов
**Что случилось:** Бюро перевело всё на MTPE (machine translation post-editing). Ставка упала с $0.10 до $0.02 за слово. Работать стало невозможно экономически.
**Хочет поделиться:** Детальный разбор того, как AI-перевод разрушает профессию, даже если качество хуже.
**Нужный интерактив:** Объединение переводчиков в группу, коллективное письмо клиентам, петиции.
**Нашла ли работу:** Нет, ищет уже 11 месяцев, рассматривает смену профессии.

---

### Сценарий 4: Райан, junior-разработчик, 24 года, Остин
**Компания:** SaaS-стартап, ~50 сотрудников
**Что случилось:** Компания объявила «AI-first strategy». Уволили 80% junior-разработчиков, оставив сениоров, которые используют Copilot.
**Хочет поделиться:** Проблема entry-level рабочих мест — куда идти джуниорам, если AI закрывает первую ступень карьерной лестницы.
**Нужный интерактив:** Менторство от сениоров, которые тоже потеряли работу, группа «Джуниоры без шансов».
**Нашёл ли работу:** Устроился техником в дата-центр, зарплата на 40% ниже.

---

### Сценарий 5: Наталья, HR-менеджер, 38 лет, Киев
**Компания:** Международная IT-компания
**Что случилось:** Компания внедрила AI-HR систему для скрининга резюме, онбординга и даже для проведения первичных интервью. HR-отдел из 12 человек сократили до 3.
**Хочет поделиться:** Как AI в HR создаёт ужасный опыт для кандидатов. Хочет показать внутреннюю кухню.
**Нужный интерактив:** Дискуссия «Этично ли AI в HR?», возможность рейтинговать компании по качеству HR.
**Нашла ли работу:** Работает HR-консультантом на фрилансе.

---

### Сценарий 6: Марк, голосовой актёр, 45 лет, Лос-Анджелес
**Компания:** Фриланс (студии аудиокниг, рекламные агентства)
**Что случилось:** 70% клиентов из аудиокниг перешли на ElevenLabs и аналоги. Рекламные ролики тоже стали озвучивать AI.
**Хочет поделиться:** Как AI-голоса обучаются на его реальных записях без разрешения. Юридический аспект.
**Нужный интерактив:** Юридический раздел с шаблонами исков, группа «Голосовые актёры против AI», петиция.
**Нашёл ли работу:** Частично — озвучивает live-события и подкасты, доход упал на 60%.

---

### Сценарий 7: Анна, контент-стратег, 31 год, Берлин
**Компания:** Крупный немецкий издательский дом
**Что случилось:** Редакция использовала AI для генерации черновиков статей. Редакторов оставили, но авторов и стратегов уволили. Качество контента упало.
**Хочет поделиться:** Парадокс — компания потеряла читателей после перехода на AI, но не наняла людей обратно.
**Нужный интерактив:** Раздел «AI-откаты» — истории компаний, которые пожалели о замене людей.
**Нашла ли работу:** Да, в PR-агентстве, но с понижением.

### Сценарий 8: Дмитрий, QA-инженер, 29 лет, Новосибирск
**Компания:** Fintech-стартап
**Что случилось:** Компания заменила ручное тестирование на AI-автотесты. QA-команда из 8 человек сокращена до 2 автоматизаторов.
**Хочет поделиться:** Как AI-автотесты пропускают edge-cases и реальные баги, но менеджмент этого не видит.
**Нужный интерактив:** Техническая дискуссия «AI в QA: работает или нет?», таблица багов, пропущенных AI.
**Нашёл ли работу:** Переучился на DevOps, работу ищет 4 месяца.

### Сценарий 9: Сара, иллюстратор детских книг, 37 лет, Портленд
**Компания:** Фриланс (несколько издательств)
**Что случилось:** Два из трёх издательств-заказчиков перешли на AI-иллюстрации. Третье держится, но платит меньше.
**Хочет поделиться:** Портфолио «до и после» — сравнение её работ с AI-заменами. Хочет показать разницу в качестве.
**Нужный интерактив:** Галерея «Человек vs AI» — где пользователи выкладывают пары работ для сравнения. Голосование.
**Нашла ли работу:** Преподаёт рисование в школе, фриланс почти на нуле.

---

### Сценарий 10: Алексей, финансовый аналитик, 33 года, Москва
**Компания:** Крупный банк
**Что случилось:** AI-модели стали генерировать аналитические отчёты, которые раньше писал отдел из 20 человек. Оставили 5 сениоров для ревью.
**Хочет поделиться:** Как AI-отчёты содержат «галлюцинации» — ложные данные, которые выглядят убедительно, но опасны для финансовых решений.
**Нужный интерактив:** Раздел «AI-провалы» — база кейсов, когда AI-замена привела к проблемам. Статистика.
**Нашёл ли работу:** Да, в другом банке, но с акцентом на «human oversight of AI».

### Сценарий 11: Кевин, оператор call-центра, 26 лет, Манила
**Компания:** Аутсорсинговая компания (обслуживала американских клиентов)
**Что случилось:** Клиент (крупный телеком) заменил call-центр на AI-чатботов. Офис из 200 человек закрыли.
**Хочет поделиться:** Масштаб увольнений на Филиппинах, где call-центры — основа экономики целых районов.
**Нужный интерактив:** Географическая группа «Юго-Восточная Азия», статистика по аутсорсинговым хабам.
**Нашёл ли работу:** Нет, рынок call-центров схлопывается.

### Сценарий 12: Лиза, журналист местной газеты, 41 год, Колумбус (Огайо)
**Компания:** Региональная газета
**Что случилось:** Редакция начала использовать AI для написания местных новостей (спорт, погода, полицейские сводки). Трёх журналистов уволили.
**Хочет поделиться:** Как AI-журналистика подрывает местную прессу и демократию. Конкретные примеры ошибок.
**Нужный интерактив:** Раздел «Медиа в опасности», карта закрывшихся редакций.

**Хочет поделиться:** Клиенты тратят часы на AI-чатботов вместо того, чтобы прочитать чистую документацию. Скрытые расходы.
**Нужный интерактив:** Калькулятор «Скрытая цена AI-замены» — инструмент для подсчёта реальных потерь компании.

### Сценарий 14: Мария, фотограф-фрилансер, 35 лет, Мадрид
**Компания:** Фриланс (рекламные агентства, стоковые платформы)

**Хочет поделиться:** Как авторское право на фотографии нарушается при обучении моделей.
**Нужный интерактив:** Юридический раздел, коллективный иск, маркировка «Human-made» для заказчиков.
**Нашла ли работу:** Снимает свадьбы и мероприятия — единственное, что AI не может заменить.

### Сценарий 15: Олег, специалист по вводу данных, 27 лет, Казань

**Хочет поделиться:** Как быстро рутинная работа исчезает, и что делать людям без высшего образования.
**Нужный интерактив:** Раздел «Рабочие профессии» — для людей, чья работа не требовала вуза. Курсы переобучения.
**Нашёл ли работу:** Курьер, зарплата сопоставимая, но без перспектив.

### Сценарий 16: Джулия, композитор для рекламы, 32 года, Нэшвилл

### Сценарий 17: Андрей, SEO-специалист, 30 лет, Варшава
**Компания:** Digital-агентство
**Что случилось:** AI-инструменты (ChatGPT, Jasper, Surfer) позволили одному человеку делать работу пятерых. Трёх SEO-шников уволили.

## ЧАСТЬ 5: КЛЮЧЕВЫЕ МОТИВАЦИИ ПОЛЬЗОВАТЕЛЕЙ

Из 30 сценариев выявляются следующие основные причины, по которым люди хотели бы прийти на такую платформу:

### Почему люди придут на сайт:
1. **Быть услышанным** — поделиться историей, когда кажется, что никому нет дела
2. **Найти «своих»** — убедиться, что ты не один, найти людей в такой же ситуации
3. **Предупредить других** — дать сигнал людям в своей профессии
4. **Получить практическую помощь** — юридические советы, курсы, вакансии
5. **Собрать доказательства** — показать, что AI-замена часто не работает
6. **Привлечь внимание** — сделать проблему видимой для СМИ и политиков
7. **Дать отпор** — коллективные действия, петиции, иски
8. **Обновить свою историю** — рассказать, что нашёл работу, что стало лучше

### Топ причин увольнения (из сценариев):
1. AI «достаточно хорош» при значительно меньшей стоимости
2. Компания объявила «AI-first strategy» и сократила штат
3. Клиенты/заказчики ушли к AI-инструментам (для фрилансеров)
4. Рутинные задачи автоматизированы, а нерутинных недостаточно для полной ставки
5. AI позволяет одному сотруднику делать работу пятерых
6. Менеджмент верит в AI-хайп и не видит реального качества
7. Целая индустрия схлопнулась (call-центры, контент-фабрики)

### Смогли ли найти работу (из 30 сценариев):
- **Нашли работу:** 18 из 30 (60%)
- **В поиске:** 7 из 30 (23%)
- **Частично / фриланс:** 5 из 30 (17%)
- **В той же сфере:** 8 из 18 нашедших
- **Сменили сферу:** 10 из 18 нашедших
- **Среднее время поиска:** ~6 месяцев
- **Зарплата на новом месте ниже:** У 12 из 18 нашедших

# ЧАСТЬ 2: ТЕХНИЧЕСКАЯ СПЕЦИФИКАЦИЯ ПЛАТФОРМЫ
## Роли, регистрация, безопасность, Hero-страница, админка, форум

> Результат брейншторма 5 итераций × 5 агентов:
> **Агент 1 — Архитектор** (системная архитектура, стек, масштабирование)
> **Агент 2 — Security Engineer** (безопасность, защита, compliance)
> **Агент 3 — UX/Product Designer** (интерфейсы, флоу, hero-страница)
> **Агент 4 — Community Manager** (роли, модерация, форум, контент-политика)
> **Агент 5 — Growth / Data Analyst** (метрики, каунтеры, тикеры, аналитика)

## 1. СИСТЕМА РОЛЕЙ

### 1.1 Иерархия ролей (от минимальных прав к максимальным)

```
┌─────────────────────────────────────────────────────┐
│  SUPERADMIN (Владелец платформы)                     │
│  ├── ADMIN (Полный доступ к админке)                 │
│  │   ├── MODERATOR (Модерация контента и форума)     │
│  │   │   ├── VERIFIED_USER (Подтверждённый автор)    │
│  │   │   │   ├── USER (Зарегистрированный)           │
│  │   │   │   │   └── GUEST (Незарегистрированный)    │
│  │   │   │   └── EXPERT (Юрист / HR / Карьерный коуч)│
│  │   │   └── COMMUNITY_LEAD (Лидер группы)           │
│  │   └── JOURNALIST (Аккредитованный журналист)      │
│  └── DATA_ANALYST (Доступ к аналитике и API)         │
└─────────────────────────────────────────────────────┘
```

### 1.2 Детальное описание каждой роли

#### GUEST (Гость)
- Просмотр hero-страницы, каунтеров, тикеров
- Чтение опубликованных историй (лимит: 5 историй в день без регистрации)
- Просмотр статистики и рейтингов компаний
- Просмотр форума (без возможности писать)
- Видит баннер «Зарегистрируйтесь, чтобы поделиться своей историей»
- **Не может:** писать, комментировать, голосовать, скачивать данные

#### USER (Зарегистрированный пользователь)
- Всё, что может GUEST, плюс:
- Подача своей истории (проходит модерацию)
- Комментирование историй (с лимитом: 20 комментариев в день)
- Голосование «Мне тоже так» / «Поддерживаю»
- Участие в форуме (создание тем, ответы)
- Подписка на группы по профессиям и темам
- Обновление своей истории (статус: «нашёл работу»)
- Настройка уведомлений
- Личные сообщения другим пользователям (с лимитом: 10 в день)
- **Не может:** публиковать без модерации, видеть флаги других пользователей

#### VERIFIED_USER (Верифицированный пользователь)
- Всё, что может USER, плюс:
- Прошёл верификацию (подтвердил, что реально потерял работу — например, документ об увольнении, LinkedIn-профиль, скриншот письма с замазанными данными)
- Его истории получают бейдж «Verified Story» ✓
- Публикации проходят ускоренную модерацию (или автоматически)
- Может загружать документы и доказательства
- Повышенные лимиты на комментарии (50/день) и сообщения (30/день)
- Может создавать опросы в форуме
- **Не может:** модерировать, видеть админку

#### EXPERT (Эксперт — юрист, HR, карьерный коуч)
- Всё, что может VERIFIED_USER, плюс:
- Верифицированная экспертиза (диплом, лицензия, подтверждение опыта)
- Бейдж «Expert: [специализация]» 🎓
- Может отвечать в разделе «Спросите эксперта»
- Может создавать гайды и ресурсы (проходят модерацию)
- Может проводить AMA-сессии (Ask Me Anything)
- Профиль с ссылками на практику/сайт
- **Не может:** модерировать контент других пользователей

#### COMMUNITY_LEAD (Лидер сообщества / группы)
- Всё, что может VERIFIED_USER, плюс:
- Назначается админом для конкретной группы (напр. «Копирайтеры», «SEO», «Call-центры»)
- Может закреплять и удалять посты в своей группе
- Может создавать события и мероприятия в группе
- Может приглашать экспертов в группу
- Видит базовую статистику активности группы
- **Не может:** банить пользователей платформы (только эскалация к модератору)

#### JOURNALIST (Аккредитованный журналист)
- Всё, что может USER, плюс:
- Верификация через редакцию или пресс-карту
- Бейдж «Press» 📰
- Доступ к агрегированной статистике для публикаций
- Может запрашивать интервью у пользователей через платформу (пользователь решает — принять или отклонить)
- Может встраивать виджеты со статистикой платформы в свои статьи
- Ограниченный API-доступ (только агрегированные данные, без персональных)
- **Не может:** видеть личные данные пользователей, скачивать базу

#### MODERATOR (Модератор)
- Всё, что может VERIFIED_USER, плюс:
- Одобрение / отклонение историй в очереди модерации
- Редактирование историй (удаление персональных данных третьих лиц, ненормативной лексики)
- Скрытие / удаление комментариев с нарушениями
- Мут пользователей (временный бан на комментирование: 1ч / 24ч / 7д)
- Эскалация сложных случаев к Admin
- Просмотр истории модерации и флагов
- Назначение тегов и категорий
- **Не может:** банить перманентно, менять роли, видеть платёжные данные, менять настройки платформы

#### ADMIN (Администратор)
- Всё, что может MODERATOR, плюс:
- Полный доступ к админ-панели
- Управление пользователями: бан, разбан, смена ролей (кроме SUPERADMIN)
- Управление контентом: удаление любых историй, комментариев, тем
- Управление категориями, тегами, группами
- Просмотр полной аналитики и логов
- Настройка автомодерации (фильтры, стоп-слова, AI-проверка спама)
- Управление SEO-настройками и метаданными
- Экспорт данных (агрегированных)
- Управление новостным разделом
- **Не может:** удалить SUPERADMIN, изменить ключевые настройки безопасности, доступ к серверу

#### SUPERADMIN (Владелец)
- Всё, что может ADMIN, плюс:
- Управление ADMIN-аккаунтами
- Доступ к настройкам безопасности и инфраструктуры
- Управление биллингом и подписками
- Доступ к бэкапам и восстановлению
- Изменение политик конфиденциальности
- Доступ к серверным логам
- Управление API-ключами
- Единственный, кто может удалить платформу

#### DATA_ANALYST (Аналитик данных)
- Доступ к агрегированной аналитике через админку
- API-доступ с расширенными квотами
- Может создавать кастомные отчёты и дашборды
- Может экспортировать анонимизированные данные
- **Не может:** видеть личные данные, модерировать контент, менять настройки

## 2. РЕГИСТРАЦИЯ И АВТОРИЗАЦИЯ

### 2.1 Готовые блоки и сервисы для интеграции

#### Рекомендуемый стек авторизации:

| Компонент | Рекомендация (Primary) | Альтернатива | Почему |
|-----------|----------------------|-------------|--------|
| **Auth-сервис** | **Clerk** | Auth0, Supabase Auth | Clerk: встроенные компоненты UI, MFA из коробки, webhooks, роли, лучший DX для Next.js |
| **OAuth-провайдеры** | Google, Apple, LinkedIn | GitHub, Microsoft | LinkedIn критичен для верификации профессии |
| **Email-сервис** | **Resend** | SendGrid, Postmark | Resend: современный API, React Email, отличная доставляемость |
| **SMS/OTP** | **Twilio Verify** | Vonage, AWS SNS | Twilio: стандарт индустрии, глобальное покрытие |
| **CAPTCHA** | **Cloudflare Turnstile** | hCaptcha, reCAPTCHA v3 | Turnstile: бесплатный, privacy-first, не раздражает пользователей |
| **Rate Limiting** | **Upstash Redis** | Redis Cloud, Cloudflare | Upstash: serverless, Edge-ready, встроенный rate limiter |
| **WAF** | **Cloudflare WAF** | AWS WAF, Fastly | Cloudflare: бесплатный тир, DDoS-защита включена |

### 2.2 Флоу регистрации (пошагово)

#### Способ 1: Email + пароль
```
[Шаг 1] Email → проверка формата → отправка OTP-кода
[Шаг 2] Ввод OTP (6 цифр, срок жизни 10 мин, 3 попытки)
[Шаг 3] Создание пароля (мин. 12 символов, требования ниже)
[Шаг 4] Базовый профиль (имя/псевдоним, страна, профессия)
[Шаг 5] Опционально: включить 2FA (TOTP через Authenticator)
[Шаг 6] Согласие с правилами платформы + Privacy Policy
→ Аккаунт создан, роль USER
```

#### Способ 2: OAuth (Google / Apple / LinkedIn)
```
[Шаг 1] Клик «Войти через Google/Apple/LinkedIn»
[Шаг 2] OAuth flow → получение email, имени, фото
[Шаг 3] Если LinkedIn: автоматическое получение профессии и компании
[Шаг 4] Дозаполнение профиля (страна, псевдоним)
[Шаг 5] Согласие с правилами
→ Аккаунт создан, роль USER
```

#### Способ 3: Анонимная история (без полной регистрации)
```
[Шаг 1] Клик «Поделиться историей анонимно»
[Шаг 2] Только email (для уведомлений о модерации)
[Шаг 3] Заполнение формы истории
[Шаг 4] Решение CAPTCHA (Turnstile)
[Шаг 5] Согласие с правилами
→ История уходит на модерацию, аккаунт не создаётся
→ Баннер: «Создайте аккаунт, чтобы обновлять историю и общаться»
```

### 2.3 Верификация (переход USER → VERIFIED_USER)

```
[Шаг 1] Пользователь нажимает «Верифицировать мою историю»
[Шаг 2] Загружает один из документов:
         □ Письмо об увольнении (с замазанными персданными)
         □ Скриншот уведомления от HR
         □ LinkedIn-профиль с подтверждением смены работы
         □ Справка из службы занятости
         □ Выписка из трудовой книжки (для СНГ)
[Шаг 3] Модератор проверяет документ (SLA: 48 часов)
[Шаг 4] Документ удаляется после проверки (не хранится!)
→ Бейдж ✓ Verified на профиле и историях
```

### 2.4 Требования к паролю

```
Минимум 12 символов
Хотя бы 1 заглавная буква
Хотя бы 1 строчная буква
Хотя бы 1 цифра
Хотя бы 1 спецсимвол (!@#$%^&*)
Проверка по базе утёкших паролей (Have I Been Pwned API)
Запрет повторения последних 5 паролей
Принудительная смена пароля при подозрительной активности
Zxcvbn-оценка ≥ 3 из 4 (проверка на словарные и предсказуемые пароли)
```

### 3.2 Защита от основных векторов атак

#### SQL Injection
- ORM (Prisma / Drizzle) — параметризованные запросы по умолчанию
- Запрет raw SQL в пользовательском коде
- Row Level Security (RLS) в PostgreSQL
- Prepared statements для всех database-вызовов

#### XSS (Cross-Site Scripting)
- Content Security Policy (CSP) headers с strict-dynamic
- Санитизация всего пользовательского ввода через DOMPurify
- React — автоэкранирование по умолчанию
- Запрет dangerouslySetInnerHTML
- HttpOnly + Secure + SameSite=Strict cookies
- Subresource Integrity (SRI) для внешних скриптов

#### CSRF (Cross-Site Request Forgery)
- SameSite=Strict cookies
- CSRF-токены для всех state-changing операций
- Double Submit Cookie Pattern
- Origin и Referer header validation

#### Brute Force & Credential Stuffing
- Rate limiting: 5 попыток входа → 15 минут блокировки → экспоненциальный backoff
- CAPTCHA (Turnstile) после 3 неудачных попыток
- Account lockout после 10 попыток → разблокировка только через email
- Проверка паролей по Have I Been Pwned
- Уведомление пользователя о входе с нового устройства/IP

#### DDoS
- Cloudflare DDoS Protection (L3/L4/L7)
- Rate limiting на API: 100 req/min для AUTH, 300 req/min для READ, 50 req/min для WRITE
- Challenge page для подозрительного трафика
- Auto-scaling инфраструктуры
- Graceful degradation: при атаке отключаются тяжёлые функции (поиск, аналитика)

#### Data Breach Prevention
- Шифрование at rest (AES-256) для базы данных
- Шифрование in transit (TLS 1.3)
- PII хранится отдельно от контента (отдельная таблица, отдельный encryption key)
- Верификационные документы: не хранятся после проверки (processed → deleted)
- Email-адреса хешируются для поиска, расшифровываются только при необходимости
- Бэкапы зашифрованы и хранятся в другом регионе
- Key rotation каждые 90 дней

#### Bot Protection
- Cloudflare Bot Management
- Turnstile CAPTCHA на формах
- Honeypot fields в формах (скрытые поля, которые заполняют только боты)
- JavaScript challenge для API-эндпоинтов
- Device fingerprinting для обнаружения автоматизации
- Ограничение скорости создания аккаунтов с одного IP (макс. 3 в час)

### 3.3 Privacy & Compliance

| Требование | Реализация |
|------------|-----------|
| GDPR | Право на удаление (полное стирание данных за 72 часа), экспорт данных, согласие на обработку |
| Анонимность | Возможность публикации под псевдонимом, IP не привязывается к истории |
| Документы верификации | Удаляются сразу после проверки, не индексируются, не кешируются |
| Логи | Логи доступа хранятся 90 дней, затем удаляются |
| Третьи лица | AI-автоудаление имён третьих лиц из историй перед публикацией |
| Cookie Consent | Минимальные cookies (только essential), отказ от tracking cookies |
| Data Minimization | Собираем только то, что необходимо, ничего лишнего |
| Breach Notification | Уведомление пользователей в течение 72 часов при утечке |

### 3.4 Аудит и мониторинг

```
Каждое действие логируется:
{
  timestamp: "2026-02-11T14:30:00Z",
  actor_id: "user_abc123",
  actor_role: "MODERATOR",
  action: "APPROVE_STORY",
  target_id: "story_xyz789",
  ip_hash: "sha256:...",
  user_agent: "...",
  result: "SUCCESS"
}

Алерты (мгновенные уведомления команде):
- 10+ неудачных входов с одного IP за 5 минут
- Админ-действие из нового IP/страны
- Массовое удаление контента
- Аномальный всплеск регистраций (>100/час)
- Попытка SQL injection или XSS в input-полях
- Попытка доступа к admin-эндпоинтам без соответствующей роли
```

### 4.2 Каунтеры — откуда берутся данные

| Каунтер | Источник данных | Обновление |

| «Историй на сайте» | COUNT из БД (только опубликованные) | Реальное время (WebSocket) |
| «Нашли работу» | COUNT из БД (stories WHERE found_job = true) | Реальное время |
| «Компаний в базе» | COUNT DISTINCT company_name из историй | Каждый час |

**Технически:**
- Каунтеры хранятся в Redis для мгновенного отображения
- Анимация при первой загрузке: числа «накручиваются» от 0 (countUp.js)
- WebSocket-push при добавлении новой истории (число мигает и увеличивается)
- Tooltip при наведении: «Данные основаны на X проверенных историях + Y публичных источников»

### 6.4 Антиспам и антиабьюз

```
Механизмы:
1. Rate limiting:
   - USER: 5 тем в день, 20 комментариев в день
   - VERIFIED_USER: 10 тем, 50 комментариев
   - Новый аккаунт (< 7 дней): 2 темы, 5 комментариев

2. Карма-система:
   - Новый пользователь стартует с karma = 0
   - +1 за каждую реакцию на его посты
   - +5 за верификацию
   - +10 за историю, прошедшую модерацию
   - -10 за удалённый модератором пост
   - При karma < -20: автоматический мут, эскалация к модератору

3. Shadow-модерация:
   - Пользователь с karma < -10: его посты видны только ему
   - Он не знает, что его заглушили
   - Модератор решает: разбанить или забанить

4. Cooldown:
   - Между созданием тем: минимум 5 минут
   - Между комментариями: минимум 30 секунд
   - Между личными сообщениями: минимум 60 секунд

5. Anti-brigading:
   - Обнаружение массового флагирования одного пользователя
   - Обнаружение множественных аккаунтов (device fingerprint + IP correlation)
   - Alert при внезапном всплеске регистраций с одного IP-диапазона
```

| **Auth** | Clerk | MFA, OAuth, роли, webhooks, UI-компоненты |
| **Database** | PostgreSQL (Neon или Supabase) | RLS, полнотекстовый поиск, JSON-поля |

| **Кеширование** | Upstash Redis | Каунтеры, rate limiting, sessions, кеш |

| **Monitoring** | Sentry + Better Uptime | Ошибки, performance, uptime |
| **CDN/WAF** | Cloudflare | DDoS protection, caching, bot management |

## 10. ИТОГИ БРЕЙНШТОРМА ПО АГЕНТАМ

### Агент 1 (Архитектор): Ключевой вывод
> Платформа должна быть serverless-first с edge-рендерингом для минимальных задержек по всему миру. PostgreSQL + Redis + Meilisearch покрывают 100% потребностей. Clerk экономит 2-3 месяца разработки auth.

### Агент 2 (Security): Ключевой вывод
> Главные риски: утечка PII уволенных людей и DDoS-атаки от компаний, недовольных рейтингом. Разделение PII в отдельную зашифрованную таблицу, документы верификации не хранить, Cloudflare WAF обязательно с первого дня.

### Агент 3 (UX/Product): Ключевой вывод
> Hero-страница должна вызывать два чувства одновременно: «я не один» (каунтеры, истории) и «есть надежда» (тикер нашедших работу). Форма подачи истории — максимум 3 шага, иначе конверсия упадёт на 60%.

### Агент 4 (Community): Ключевой вывод
> Без жёсткой модерации с первого дня платформа превратится в токсичное пространство. Нужна карма-система, shadow-бан для троллей и обязательные COMMUNITY_LEAD для каждой профессиональной группы. Запретить дизлайки — только позитивные реакции.

### Агент 5 (Growth/Data): Ключевой вывод
> Каунтеры — главный виральный элемент. Скриншот hero-страницы с числом «47,832 уволено из-за AI» будет расшариваться в соцсетях. API для журналистов — бесплатный рост. Тикер «лидеров увольнений» вызовет PR-давление на компании, что привлечёт СМИ.

### 2.1 Многоуровневая система «Один человек = один аккаунт»

```
УРОВЕНЬ 1: EMAIL (обязательный)
┌─────────────────────────────────────────────────────┐
│ • Один email = один аккаунт (UNIQUE constraint)     │
│ • Email подтверждается через OTP (не просто ссылка) │
│ • Запрет временных email (mailinator, guerrilla и   │
│   др.) — используем библиотеку disposable-email-    │

│ • Запрет email-alias'ов:                            │
│   user+spam@gmail.com → нормализуется в             │
│   user@gmail.com перед проверкой уникальности       │
│ • Запрет точек в gmail:                             │
│   u.s.e.r@gmail.com → user@gmail.com                │
│ • Rate limit: макс. 3 OTP-запроса в час на один IP  │
│ • Email хешируется (SHA-256) для проверки            │
│   уникальности, шифруется (AES-256) для хранения    │
└─────────────────────────────────────────────────────┘
            │
            ▼

┌─────────────────────────────────────────────────────┐

┌─────────────────────────────────────────────────────┐
```

| 2 | **Астротурфинг от компаний** | PR-агентство создаёт 50 аккаунтов с позитивными историями: «В компании X меня не уволили, всё отлично!» | Корреляция fingerprint/IP + анализ тона (слишком позитивные истории о конкретной компании → флаг) + обязательный телефон | ✅ Anomaly detection |
| 3 | **Клевета** | Ложная история об увольнении из компании, в которой человек не работал | Верификация (документы) + пометка «Unverified» на непроверенных историях + право компании оспорить | ⚠️ Частично (верификация ручная) |
| 4 | **Доксинг** | Публикация ФИО менеджера, его телефона, адреса в истории | AI-детектор PII (имена, телефоны, адреса, email) → автоматическая замена на [REDACTED] перед публикацией | ✅ NER + regex |
| 5 | **Victim blaming в комментариях** | «Сам виноват, надо было учить AI» «Копирайтеры не нужны, смирись» | AI-детектор токсичности + карма-система + report button + shadow-ban | ✅ Toxicity classifier |
| 6 | **Раскрытие NDA** | Пользователь публикует внутренние данные компании, защищённые NDA | Дисклеймер в форме + модерация + право на takedown по запросу от компании (с проверкой) | ⚠️ Частично (предупреждение автоматическое, решение ручное) |
| 7 | **Эмоциональные манипуляции** | Фейковая «трагическая» история для сбора донатов или сочувствия | AI-детектор фейков + верификация + отсутствие функции денежных переводов на платформе | ⚠️ Частично |
| 8 | **Скрейпинг данных** | Бот собирает все истории + метаданные для перепродажи или для компаний | Rate limiting + anti-scraping headers + Cloudflare Bot Management + динамическая пагинация | ✅ Автоматически |
| 9 | **SEO-спам** | Ссылки в историях и комментариях на внешние сайты | Все внешние ссылки: nofollow + автоматическая проверка URL по базе спам-доменов + лимит 2 ссылки на пост | ✅ URL classifier |
| 10 | **Координированная атака** | Группа аккаунтов массово флагирует легитимную историю, чтобы её убрали | Anti-brigading: если > 10 флагов за 1 час от аккаунтов младше 30 дней → игнорировать + расследовать флаггеров | ✅ Anomaly detection |
| 11 | **Шантаж компании** | «Уберите мою историю, или я напишу ещё хуже» | Модерация + запрет редактирования после модерации (только через запрос) + аудит всех изменений | ⚠️ Ручное |
| 12 | **Межпользовательский конфликт** | Два уволенных спорят в комментариях, переходя на личности | Auto-lock темы при > 5 зафлаженных комментариях + cooldown для участников конфликта | ✅ Частично |
| 13 | **Самоповреждение** | Человек в отчаянии пишет о суицидальных мыслях | AI-детектор кризисного контента → мгновенный alert модератору + автоматический показ горячей линии помощи + обученные модераторы | ✅ AI alert + ручной ответ |
| 14 | **Политизация** | Превращение платформы в политическую площадку «AI — заговор демократов/республиканцев» | Правила: только личные истории, без политической агитации. AI-детектор политического контента → модерация | ⚠️ Сложно автоматизировать |
| 15 | **Манипуляция каунтерами** | Создание 100 аккаунтов с фейковыми историями для раздувания числа «уволенных» | Каунтеры считают только Verified stories + отдельная метка «включая неверифицированные» | ✅ Логика на бэкенде |

### 5.2 Конкретные API и сервисы для AI-модерации

| Задача | Сервис | Стоимость | Точность |
|--------|--------|-----------|----------|
| Toxicity / Hate Speech | OpenAI Moderation API | Бесплатно | ~95% |
| Детальная токсичность | Perspective API (Google Jigsaw) | Бесплатно до 1 QPS | ~92% |
| PII Detection | Presidio (Microsoft, open-source) | Бесплатно (self-hosted) | ~90% |
| AI-generated text | GPTZero API | $10/мес (10K запросов) | ~85% |
| Spam classification | Custom model на базе DistilBERT | Бесплатно (self-hosted) | ~93% |
| Crisis detection | Custom keyword model + OpenAI | ~$5/мес | ~88% |
| Image moderation (если загрузка изображений) | AWS Rekognition или Sightengine | $1/1000 изображений | ~97% |

### 5.3 Обучение модели на контенте платформы

```
Месяц 1-3: Используем готовые API (OpenAI Moderation, Perspective)
           + ручную модерацию 100% контента

Месяц 3-6: Собираем размеченные данные из ручной модерации
           (одобрено / отклонено + причина)
           Минимум 5,000 размеченных примеров

Месяц 6+:  Fine-tune собственной модели на данных платформы
           DistilBERT или DeBERTa для классификации
           Деплой через Hugging Face Inference
           Готовые API остаются как fallback и проверка
```

### 7.2 Техническая реализация Telegram-бота

| Компонент | Решение |
|-----------|---------|

| Хостинг | Тот же сервер (Vercel Edge Functions или отдельный worker на Railway) |

| Inline-кнопки | Для модерации: Approve/Reject прямо из Telegram |
| Rate limiting | Макс. 1 история в день через бот, 10 сообщений в минуту |
| Антиспам | Проверка Telegram user_id на уникальность (один Telegram = один аккаунт) |

## 8. ПРОБЛЕМЫ, О КОТОРЫХ ЧАСТО ЗАБЫВАЮТ

### 8.1 Эмоциональная нагрузка

```
ПРОБЛЕМА:
Платформа собирает истории людей в тяжёлой жизненной ситуации.
Это создаёт:
- Эмоциональное выгорание модераторов
- Риск «кризисных» постов (суицидальные мысли)
- Негативный эмоциональный фон может отталкивать новых пользователей

РЕШЕНИЯ:

1. Модераторы:
   - Ротация: никто не модерирует > 4 часов подряд
   - Ограничение: макс. 30 историй на модерацию в день на человека
   - Обязательный перерыв после кризисного контента
   - Доступ к психологической поддержке (если команда волонтёров)

2. Кризисный контент:
   - AI-детектор → мгновенный алерт
   - Автоматический баннер под постом:
     «Если вы переживаете трудный период:
      🇷🇺 Телефон доверия: 8-800-2000-122 (бесплатно)
      🇺🇸 988 Suicide & Crisis Lifeline
      🌍 findahelpline.com»
   - НЕ удалять пост — человеку важно быть услышанным
   - Модератор пишет поддерживающий комментарий

3. Баланс тона:
   - Тикер «Нашли работу» на главной — обязателен
   - Раздел «Истории успеха» всегда вверху
   - Еженедельная рассылка с позитивными обновлениями
   - Пометка на историях, где автор позже нашёл работу:
     «Обновление: автор нашёл работу через 4 месяца! ✅»
```

### 8.2 Масштабирование модерации

```
ПРОБЛЕМА:
При росте до 100+ историй в день ручная модерация не справится.

РЕШЕНИЕ — ТРЁХСТУПЕНЧАТАЯ МОДЕРАЦИЯ:

Ступень 1: AI (мгновенно)
→ Auto-approve: 40% контента (проверенные пользователи + чистые scores)
→ Auto-reject: 5% контента (явный спам, токсичность)
→ В очередь: 55% контента

Ступень 2: Community (Community Leads)
→ Лидеры профессиональных групп проверяют контент в своей области
→ Знают контекст профессии → лучше выявляют фейки
→ Покрывают 30% очереди

Ступень 3: Модераторы (команда)
→ Сложные случаи, эскалации, апелляции
→ Покрывают 25% очереди
→ Финальное решение по спорным случаям

ЦЕЛЕВЫЕ SLA:
- 80% контента: модерация < 2 часов
- 95% контента: модерация < 12 часов
- 100% контента: модерация < 48 часов
```

### 8.3 Интернационализация и культурные различия

```
ПРОБЛЕМА:
Увольнения из-за AI — глобальная проблема. Но:
- В США: NDA распространены, страх судебных исков
- В Европе: GDPR, сильная защита работников
- В СНГ: другая культура жалоб, трудовые книжки
- В Азии: call-центры — основа экономики регионов
- Разные языки, разные юридические рамки

РЕШЕНИЯ:
- Мультиязычность с 1 дня (минимум EN + RU)
- Региональные правила модерации
- Региональные юридические дисклеймеры
- Региональные ресурсы помощи (горячие линии, службы занятости)
- Региональные Community Leads
- Автоперевод историй (опционально, через DeepL API)
```

### 8.4 Манипуляция статистикой

```
ПРОБЛЕМА:
Каунтеры на главной — вирусный элемент. Но если 30% историй фейковые,
статистика бессмысленна и подрывает доверие.

РЕШЕНИЯ:
1. Два набора каунтеров:
   - «Всего историй: 12,461» — все опубликованные
   - «Верифицированных: 4,823» — только с проверкой
   - Журналисты и исследователи используют только верифицированные

2. Methodology page:
   - Прозрачное описание: как считаем, что включаем
   - Какие источники кроме пользовательских историй
   - Какие ограничения и погрешности

3. Anomaly detection:
   - Если за день историй в 5 раз больше обычного → алерт
   - Если 10+ историй о компании X за час → проверка на координацию
   - Если каунтер «уволенных» растёт быстрее каунтера «историй»
     → значит, внешний парсер работает некорректно → пауза

4. Разделение пользовательских данных и агрегированных:
   - Каунтер «уволено из-за AI» = наши данные + внешние трекеры
   - Каунтер «историй» = только наши, проверенные
   - Всегда показывать источники при наведении
```

### 8.5 Конфликт интересов с AI-компаниями

```
ПРОБЛЕМА:
Платформа критикует эффекты AI.
AI-компании (OpenAI, Google, Anthropic) могут:
- Не хотеть сотрудничать
- Блокировать API-доступ (Moderation API и т.д.)
- Оказывать PR-давление

РЕШЕНИЯ:
- Позиционирование: «Мы не против AI. Мы за людей.
  Мы документируем влияние на рабочих, а не критикуем технологию.»
- Диверсификация зависимостей: не полагаться на один AI-сервис
- Open-source модели модерации как backup (Llama, Mistral)
- Независимое финансирование (гранты, донаты, не венчурный капитал)
```
